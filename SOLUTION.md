# Поиск статей справки — описание подхода

Retrieval-этап для RAG: по короткому разговорному вопросу вернуть ранжированный
топ-10 `article_id`. Метрика — **MAP@10**. Публичный **тест = 0.622545** (нижняя
граница «популярность» = 0.32, оракул = 1.0).

Эволюция на публичном тесте: 0.5281 (гибрид dense модели и bm25 + cross-encoder reranker) → 0.5338 (гибрид еще с e5 +best_chunk
reranker) → 0.6006 (+ дообучение dense моделей) → **0.622545 (+ дообучение reranker)**.
Ключевой скачок дало супервизное LoRA-дообучение всех трёх нейросетей на размеченных
парах calibration (dense +0.067, reranker +0.022) — см. раздел 4.

Пошаговая инструкция воспроизведения — в [README.md](README.md).

## 1. Данные и их особенности (EDA — `notebooks/01_eda.ipynb`)

- Корпус — **793 статьи** (`article_id`, `title`, `body` в HTML со служебной
  разметкой: таблицы, табы, спойлеры, factoid-блоки, картинки). В корпусе есть
  **дубли статей** (173 статьи с заголовком-двойником) — учтено при майнинге негативов.
- `calibration.f` — 500 размеченных запросов; `test.f` — 500 без разметки.
  Распределения длин запросов у calibration и test совпадают → калибровке можно
  доверять как прокси.
- **Ключевые находки, определившие подход:**
  - **44% запросов имеют >1 правильной статьи** → MAP@10 требует поднимать наверх
    *все* релевантные, не одну.
  - **calibration сильно смещён**: покрывает лишь 79/793 статей, перекос в
    доставку/возвраты; «хаб»-статья `4219 «Покупателю»` — правильный ответ для 129
    из 500 запросов. Отсюда риск переобучения при подборе гиперпараметров.
  - Запросы **разговорные, с опечатками и маск-токенами** (`<MONEY>`, `<ID>`,
    `<DATE>`) → лексика запроса ≠ язык статей.
  - Между статьями есть **граф внутренних ссылок** (924 ребра внутри корпуса).

## 2. Обработка текста (`src/data.py`, `src/text.py`)

- **Очистка HTML → текст** (`html_to_text`): удаляем `script/style`, `<img>`
  заменяем его `alt`-текстом, собираем видимый текст, схлопываем пробелы. Текст
  таблиц/табов/спойлеров сохраняется. Стратегия сознательно «щадящая»: убираем
  только гарантированный не-текст.
- **Маск-токены** (`<MONEY>`, …) вырезаются — иначе превращаются в мусорные токены
  (дало небольшой прирост BM25).
- **Для BM25** (`tokenize`): нижний регистр, `ё→е`, токенизация по буквам/цифрам
  (устойчиво к слипшимся словам и пунктуации), удаление русских стоп-слов и
  «вежливых» слов поддержки. **Стемминг выключен** — на этих данных Snowball
  переусердствует и снижает метрику (проверено абляцией).
- **Для dense/reranker** (`clean_text`): естественный текст без токенизации/стемминга.
- Дополнительно извлекаются **anchor-тексты ссылок** и **id связанных статей**
  (граф); anchor'ы используются как поле в BM25.

## 3. Базовый пайплайн (retrieve → rerank)

Собран послойно, каждый слой обоснован замером на calibration (MAP@10):

| Слой | Что | MAP@10 |
|---|---|---|
| baseline | популярность (нижняя граница) | 0.3176 |
| 1 | **BM25** (Okapi, `title×3 + body + anchors`, k1=2.5, b=0.4) | 0.329 |
| 2 | dense **RoSBERTa** (`ai-forever/ru-en-RoSBERTa` 0.4B) | 0.446 |
| 2 | dense **e5-large** (`intfloat/multilingual-e5-large` 0.6B) | 0.422 |
| 3 | **3-way гибрид** BM25+RoSBERTa+e5 (RRF) | 0.5189 |
| 4 | **+ cross-encoder reranker (0.6B)** (blend, «голова» статьи) | 0.5324 |
| 4 | **+ реранк по лучшему чанку** (blend) | **0.5369** |

**Dense-ретривер (`src/retrievers/dense.py`).** Bi-encoder; статьи режутся на
пассажи ~200 слов с перекрытием 40, к каждому чанку добавляется заголовок; чанки
кодируются с префиксом `passage:`/`search_document:`, запрос — `query:`/`search_query:`;
эмбеддинги L2-нормализованы (косинус = скалярное произведение); скор статьи =
**max по её чанкам**. Эмбеддинги корпуса кешируются на диск.

**Гибрид (`src/retrievers/hybrid.py`).** Слияние BM25 + двух dense-моделей
через **Reciprocal Rank Fusion** (по рангам, без нормализации разношкальных
скоров): `score(d) = Σ w_r / (k_rrf + rank_r(d))`. Добавление e5 третьим источником
дало +0.02 к 2-way — разные модели ошибаются по-разному.

**Reranker (`src/retrievers/rerank.py`).** Cross-encoder `BAAI/bge-reranker-v2-m3` (0.6B)
скорит пары (запрос, статья) для топ-кандидатов гибрида. Два важных решения:

- **Чистый reranker ухудшает метрику** (0.43) — он демотит общие хаб-статьи,
  которые размётка считает правильными. Поэтому его ранг **подмешивается** к рангу
  гибрида через RRF, сохраняя приор гибрида.
- **Реранк по лучшему чанку** (`best_chunk=True`): cross-encoder'у подаётся не
  «голова» статьи (первые ~512 токенов), а тот dense-пассаж, что дал max при
  поиске. Для длинных статей релевантный кусок часто за пределами «головы».

В общем, мой пайплайн выглядит так:
1) Гибрид через RRF: bm25 + ROSBERTA+ e5-large.
2) Топ кандидатов гибрида переранжируются при помощи cross-encoder реранкера, но его ранжирование используется не как окончательное, а как дополнительный сигнал к уже существующему гибриду.

## 4. Супервизное дообучение (главный скачок: тест 0.534 → 0.623)

Все три нейросети дообучены **LoRA-супервизно** на размеченных парах из calibration.
Общий принцип валидации: calibration делится на **dev-400 / test-100**; обучение и
подбор всех гиперпараметров (в т.ч. веса RRF и blend) — на dev, финальный замер —
на нетронутом test-100. Данные для обучения собираются на лету из `calibration.f`.

**Dense (`scripts/train_dense.py`).**
- Данные: пары `запрос → его gt-статьи` + hard negatives (топ BM25, кроме gt, с
  фильтром дубликатов-статей). Мульти-ответные запросы дают по паре на статью.
- LoRA на Q/K/V (~0.4% параметров), MNRL-лосс, **1 эпоха** (дальше переобучение —
  видно по dev/test-сплиту: train↑, val↓).
- Эффект: e5 0.39→0.55, RoSBERTa 0.45→0.56 (single, test-100).

**Reranker (`scripts/train_reranker.py`).**
- Пары `(запрос, best_chunk статьи)` → метка 1/0 (gt / не-gt, hard negatives от
  FT-гибрида), BCE, LoRA, ~3 эпохи.
- Базовый reranker опускал общие статьи, поэтому его
  душили малым весом; после дообучения он выучил, что хаб-статьи = правильные, и его
  чистое ранжирование выросло **0.46→0.69** (test-100). Теперь в blend он получает
  большой вес (`w_rerank=5`).

**Итог на test-100 (held-out от обучения):** 3-way FT-гибрид 0.650 → + FT-reranker
**0.6845**. **Перенос на публичный тест:** 0.534 → 0.6006 (дообучение dense моделей) → **0.622545**
(дообучение реранкера). Calibration был оптимистичен (~0.06 надбавки), но реальный выигрыш
крупный и подтверждён на held-out тесте.

Финальный `answer.csv` строит `scripts/build_answer_ft.py`: BM25 + RoSBERTa-ft +
e5-ft (RRF `w_bm25=1.5, w_dense=2, k_rrf=10`) → **дообученный** cross-encoder реранкер в качестве нового сигнала в гибрид (`w_rerank=5, k_blend=5, top_n=30`).

## 5. Как проверяли качество и анализ ошибок

- Единый harness метрики **MAP@10** (`src/metrics.py`, с самопроверками) +
  пер-запросный AP; опорные границы — популярность / оракул. Оценка компонент —
  `scripts/run_eval.py --algo <name>`, полный FT-пайплайн — `scripts/eval_pipeline.py`.
- Все гиперпараметры (BM25 `k1/b`, веса RRF, blend, дообучение) подбирались на
  **dev-400** и проверялись на нетронутом **test-100**, чтобы не переобучиться под
  смещённую разметку.
- **Анализ ошибок** (`notebooks/03_hybrid_analysis.ipynb`, `04_multianswer.ipynb`):
  - запросы с AP=0: **78% — провал ранжирования** (правильная статья в топ-50, но
    ниже 10-й), лишь ~10% — провал поиска → главный резерв в переранжировании;
  - у мульти-ответных запросов «вторые» правильные статьи в основном на рангах
    11–30 → цель для reranker/перевзвешивания;
  - граф ссылок как «добор» — потолок мал (~23% чистых соседей), деприоритизирован;

## 6. Структура кода и воспроизведение

```
src/
  config.py metrics.py data.py text.py
  retrievers/  base.py popularity.py bm25.py dense.py hybrid.py rerank.py
scripts/
  run_eval.py         # оценка алгоритма на calibration: --algo <name>
  train_dense.py      # LoRA-дообучение dense (e5/RoSBERTa) на dev-400
  train_reranker.py   # LoRA-дообучение cross-encoder reranker'а
  eval_pipeline.py    # замер финального FT-пайплайна на dev-400/test-100
  build_answer_ft.py  # генерация answer.csv финальным FT-пайплайном
  build_answer.py     # генерация answer.csv (базовый пайплайн, без дообучения)
notebooks/            # 01_eda, 02_baseline, 03_hybrid_analysis, 04_multianswer
```

**Модели.** Базовые (`intfloat/multilingual-e5-large`, `ai-forever/ru-en-RoSBERTa`,
`BAAI/bge-reranker-v2-m3`) и дообученные (выложены на HuggingFace:
`iliakabanov/{russian,multillingual}-dense-retriever`, `iliakabanov/reranker-retriever`)
скачиваются с HuggingFace. Запуск — в окружении с torch/CUDA.

**Воспроизведение** (детали — в README):
```bash
python scripts/build_answer_ft.py          # answer.csv из дообученных весов (HF)
python scripts/build_answer_ft.py --local  # из локально обученных весов
```
Инференс детерминирован (сиды + cuDNN) → воспроизводимый `answer.csv`. Обучение с
нуля (dev-400) — команды в README, раздел «Вариант B».

## 7. Итог

- Прошли путь **0.32 → 0.622545** на публичном тесте по канону «baseline → анализ
  ошибок → усложнение», каждый шаг обоснован замером на честном dev/test-сплите.
- Решение **общее** (не подгонка под конкретные `query_id`): единый пайплайн для
  всех запросов, open-source модели, воспроизводимый код с фиксированными сидами.
- Главные уроки: лексики недостаточно (нужна семантика) → ансамбль моделей
  диверсифицирует ошибки → cross-encoder полезен только как поправка на смещённой
  разметке, **пока не дообучен** → супервизное дообучение всех трёх сетей на
  размеченных парах дало основной прирост (+0.089 к тесту) и перенеслось с held-out
  на публичный тест.
